- [Audio-verify borderline cuts](reference_audio_verify_borderline_cuts.md) — for "borderline" or "should I cut this?" moments in dialogue review: re-extract the disputed window with ffmpeg, re-transcribe with `large-v3` + `word_timestamps=True` + `vad_filter=False`, and check waveform energy in 50ms bins for silence boundaries. The base `turbo` Whisper used by the cut analyzer merges duplicates ("barriers, so these berries" → "berries") and drops aborted partials ("eight" → "h") — large-v3 keeps both. Decision rule: false start = first attempt incomplete + 200-400ms silence at correction → CUT; emphatic restatement = full repeated unit, no silence → KEEP. ~2 min/clip.
- [Default Whisper hides false-starts in source transcripts](reference_default_whisper_hides_false_starts.md) *(2026-05-15)*. faster-whisper with default settings silently merges false-starts into clean segments via repetition penalty. Confirmed: source audio contains *"Now of course Misty's come... Now of course Misty has to..."* but `transcripts/4.json` shows only the clean version, with a tell-tale 4.84-second `' Misty'` word at 392.04-396.88. Cut analysis based on the default transcript misses or mis-bounds cuts (proven by v6's "now of course now of course" duplicate). **Always re-transcribe source with `condition_on_previous_text=False, no_repeat_ngram_size=0, beam_size=5` before running the cut analyzer.** ~25-40 min on CPU large-v3 — acceptable one-time cost. Detection signatures: words > 1.5s long; repeated n-grams within 5s; long word followed by silence then repeat.
- [Whisper word timestamps are unreliable for cut-boundary verification](reference_whisper_word_timestamps_unreliable.md) *(2026-05-15)*. faster-whisper word timestamps can be off by 100-400ms, especially around silence boundaries. Caught when claimed "And" started at src 535.84 but actual sound onset (25ms-RMS bins) was 536.175 — a 335ms error that led me to wrongly diagnose a clip as cutting mid-word. **Auto-editor's invariant is real: clips ALWAYS start in silence.** If your Whisper word timestamps say otherwise, the timestamps are wrong, not the cut. Whisper also hallucinates phantom words (e.g. "honest", probability 0.59) at clean splice points where brief silence + sudden onset confuses its decoder — every dialogue-review delivery should include a splice-point hallucination scan (re-transcribe ±1s around each clip boundary with `condition_on_previous_text=False, no_repeat_ngram_size=0`).
- **Atomic numbered references — never split** *(2026-05-15)*. Phrases like *"Rival 2"*, *"rival number two"*, *"attempt number one"*, *"reset 29"*, *"the second gym leader"*, *"my second Pokemon"* travel as ONE unit. Cuts whose boundary lands between the noun and its number/ordinal/digit are wrong by definition — extend the cut to swallow the whole phrase OR shrink it to exclude both. Captured in `Teo Speech Style.md` §9.4 and the per-cut self-check in `mark_cut_candidates.py`. Caught when "rival number two" was lost via SRC_OVERLAP cut.
- [SRC_OVERLAP_PREV cuts remove words from BOTH copies](reference_src_overlap_cut_bug.md) — the cut analyzer's "the OTHER copy still plays" reasoning for SRC_OVERLAP cuts is wrong. `apply_cuts_to_fcpxml.py` operates on source-frame ranges, so flagging the overlap cuts those frames from the prior clip's tail (TRIM_END) AND from the next clip's head (TRIM_START) — both copies of the words are lost. Caught when "rival number two" disappeared in v5. Cut analyzer prompt now forbids SRC_OVERLAP_PREV cuts; tolerate the brief duplication.
